# Kruxia Flow Performance Investigation - November 8, 2025

## Summary

We set up comprehensive profiling infrastructure, enabled pg_stat_statements, and attempted Quick Win optimizations. **Critical findings**:
1. Extremely aggressive polling (1ms-100ms) caused complete system failure - 10x worse performance
2. **Increasing worker count from 1 to 20 had NO meaningful impact** - worker starvation was NOT the bottleneck
3. The real bottleneck is elsewhere - need CPU profiling and query analysis to identify

## Actions Taken

### ✅ Infrastructure Improvements

1. **Enabled pg_stat_statements** in Docker PostgreSQL
   - Added `shared_preload_libraries=pg_stat_statements` to docker-compose.yml
   - Now can analyze slow queries

2. **Created Profiling Scripts**
   - `scripts/profile_benchmarks.sh` - Automated profiling runner
   - `scripts/analyze_profiling_results.sh` - Automated analysis
   - `docs/profiling-quickstart.md` - User guide

3. **Connection Pool Increase**: 20 → 100 max connections
   - Status: **KEPT** (good change)

### ❌ Failed Optimization #1: Event Polling Backoff

**Attempt #1: 1ms min, 100ms max** ⚠️ TOO AGGRESSIVE
- Result: Complete system failure
- All 300 workflows timed out (30+ second timeouts)
- Throughput dropped to 3.25 wf/sec (from baseline 27 wf/sec)
- **10x WORSE performance**

**Root Cause Analysis**:
- 1ms polling created CPU spinning
- Database overwhelmed with poll queries
- Connection pool saturated
- System unable to process any actual work

**Attempt #2: 10ms min, 500ms max**
- Status: Tested with 20 workers
- More moderate than attempt #1
- Max reduced from 5000ms → 500ms
- Min kept at original 10ms (avoids CPU spinning)
- **Result: No meaningful improvement when combined with 20 workers**

### ❌ Failed Optimization #2: Worker Count Increase

**Change: 1 worker → 20 workers**
- Expected: 10-20x improvement for parallel workflows
- Actual: **No meaningful improvement**

**Results:**
- High Concurrency: 27 → 27.71 wf/sec (0.3% improvement)
- Sequential: 1.82 → 1.91 wf/sec (5% improvement)
- Parallel: 1.12 → 1.06 wf/sec (5% WORSE)
- Sustained: New baseline 2.51 wf/sec

**Conclusion:**
- Server logs confirm all 20 workers started
- But throughput unchanged
- **Worker count is NOT the bottleneck**
- Need to investigate orchestrator and database performance

## Key Findings from Profiling

### What We Confirmed

1. **Activity Queue is Fast**: 0.014ms execution time
   - Index `idx_queue_claimable` working correctly
   - NOT a bottleneck

2. **Connection Pool Was Too Small**: 20 connections inadequate for 100 concurrent workflows
   - Increased to 100 (matches concurrency)
   - Status: GOOD change, keep it

### What We Learned (The Hard Way)

3. **Event Polling is EXTREMELY Sensitive**
   - 1ms polling = system death
   - Need careful tuning, not aggressive changes
   - CPU overhead matters more than latency in this case

4. **Baseline Performance (Before Changes)**:
   - High Concurrency: 27 wf/sec, P99 5s
   - Sequential: 1.82 wf/sec, P99 6.7s
   - Parallel: 1.12 wf/sec, P99 30s

5. **After Aggressive Polling (1ms-100ms)**:
   - High Concurrency: 3.25 wf/sec, P99 32s
   - All workflows timed out
   - **Conclusion**: Polling too fast is worse than polling too slow!

## Hypothesis: The Real Bottleneck

Based on the profiling data and failed optimization, I believe the bottleneck is **NOT** event polling delay. Here's why:

1. **Aggressive polling made things worse** - If polling was the bottleneck, faster polling should have helped
2. **CPU spinning consumed resources** - The system was busy polling instead of doing work
3. **Activity queue is fast** - Database operations are not slow
4. **Workflows are completing slowly even without polling pressure** - Something else is wrong

## Hypothesis: The Real Bottleneck (Updated)

Based on testing both aggressive polling and 20x worker increase with minimal impact, the bottleneck must be:

### Primary Suspects

#### 1. Orchestrator Event Processing Lag
**Evidence:**
- Neither faster polling nor more workers helped
- P99 latencies are 5-30 seconds for simple echo activities
- Echo activity should be <10ms, but workflows take seconds

**Theory:**
- Orchestrator may be slow to:
  - Poll for workflow events
  - Evaluate workflow state
  - Schedule activities

**Test:**
```bash
# Check orchestrator timing in logs
grep "orchestrator" /tmp/kruxiaflow-profiling.log | grep -E "took|duration|elapsed"
```

#### 2. Database Query Performance
**Evidence:**
- pg_stat_statements now enabled but not yet analyzed
- Activity queue query is fast (0.014ms)
- But other queries may be slow

**Test:**
```bash
# Check slow queries
psql18 -U kruxiaflow -d kruxiaflow_profiling -c "
SELECT query, calls, mean_exec_time, stddev_exec_time
FROM pg_stat_statements
WHERE mean_exec_time > 10
ORDER BY mean_exec_time DESC
LIMIT 20;
"
```

#### 3. HTTP Round-Trip Overhead
**Evidence:**
- Built-in worker polls via HTTP API
- Each activity: HTTP poll → claim → execute → HTTP complete
- For 300 workflows × 3 activities = 900 HTTP round trips

**Test:**
```bash
# Check worker timing in logs
grep "worker" /tmp/kruxiaflow-profiling.log | grep -E "claimed|completed|took"
```

## Recommended Next Steps

### Phase 1: Analyze With Profiling Data (CRITICAL)

**Stop guessing. Get real data.**

1. **Run Profiling Session** ⏱️ 5-10 minutes
   ```bash
   # Run benchmark with profiling enabled
   ./scripts/profiling.sh --profile --test test_sequential_workflow_load
   ```

   This will capture:
   - CPU flamegraph showing where time is spent
   - PostgreSQL slow query analysis (queries > 1ms)
   - Server logs with debug output
   - All output in `profiling/results-TIMESTAMP/` directory

2. **Review Flamegraph** ⏱️ 5 minutes
   ```bash
   # Open the flamegraph
   open profiling/results-*/flamegraph.svg
   ```

   Look for:
   - **Wide bars** = hot paths (where CPU time is spent)
   - **Tall stacks** = deep call chains
   - **Flat regions** = tight loops or blocking operations

3. **Analyze Slow Queries** ⏱️ 5 minutes
   ```bash
   # View slow queries
   cat profiling/results-*/slow-queries.txt
   ```

   Look for:
   - High `mean_exec_time` (> 10ms is concerning)
   - High `calls` with slow execution (amplified impact)
   - Missing indexes (sequential scans)

### Phase 2: Fix Based on Data

**After profiling identifies the hot path:**

**IF** Orchestrator is hot path:
- Optimize state evaluation
- Reduce event polling overhead
- Cache workflow definitions

**IF** Database queries are slow:
- Add missing indexes
- Optimize query plans
- Batch operations

**IF** HTTP overhead is bottleneck:
- Direct service interface for built-in worker
- WebSocket for activity polling
- Batch activity claims

### Phase 3: Iterate and Measure

After each fix:
1. Re-run benchmarks
2. Compare to baseline
3. Document improvement
4. Profile again to find next bottleneck

## Lessons Learned

1. **Measure before optimizing** - We hypothesized event polling was the bottleneck, but aggressive polling made things worse
2. **Small changes first** - Going from 5000ms → 100ms was too aggressive; should have tried 5000ms → 1000ms → 500ms
3. **CPU matters** - Polling too fast consumes CPU that could be doing real work
4. **Connection pool was good** - That change should be kept (100 max connections)
5. **Worker count doesn't help if it's not the bottleneck** - 20 workers made no difference, proving workers weren't starved
6. **Need actual profiling data** - Flamegraph and query analysis are CRITICAL - stop guessing!
7. **Test one variable at a time** - Combining polling changes + worker count made it harder to isolate effects
