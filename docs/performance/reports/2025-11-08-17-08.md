Comprehensive benchmark suite with full profiling across all four test scenarios. System shows good performance on high concurrency workloads but struggles with parallel workflows.

## Identified Bottlenecks

### 1. Event Polling Latency (CRITICAL)

**Evidence**:
- Event poll query degrades from 0.25ms to 2.37ms over time
- 386 event polls in 60s sustained test = ~6.4 polls/sec
- With 2.37ms average × 386 polls = 915ms spent just querying events
- Actual time between polls is much higher (orchestrator backoff)

**Root Cause**:
- `SELECT ... WHERE c.last_event_id IS NULL OR e.id > c.last_event_id`
- As event table grows, query becomes slower
- No cleanup of old events

**Fix Options**:
1. **Index on workflow_events(id)** - Already partially covered by PK
2. **Partition event table** by timestamp (archive old events)
3. **Reduce polling backoff** - Currently 10ms-5s, reduce to 1ms-100ms
4. **Batch event reads** - Read more events per poll

**Expected Impact**: 5-10× improvement

### 2. Parallel Activity Scheduling (CRITICAL)

**Evidence**:
- 12% failure rate on parallel workflows
- P99 = 30,336ms (timeout) vs P50 = 887ms
- 1,187 activity deletes for 500 expected activities (2.4× overhead)

**Root Cause**:
- Orchestrator may not be scheduling all parallel activities at once
- OR workers are not claiming activities fast enough
- OR some dependency evaluation bug

**Fix Options**:
1. **Debug orchestrator**: Add logging to show when all parallel activities are scheduled
2. **Increase worker polling frequency** - Reduce idle time
3. **Fix dependency evaluation** - Ensure all parallel activities marked as ready simultaneously

**Expected Impact**: 100% success rate, 20-50× improvement in P99

### 3. Database Overhead (MINOR)

**Evidence**:
- DELETE activity_queue: 120-807ms per test (1-6% of duration)
- UPDATE heartbeats: 108-567ms per test (1-3% of duration)

**Root Cause**:
- Immediate deletion of completed activities adds overhead
- Heartbeat updates may be unnecessary for short activities

**Fix Options**:
1. **Batch delete**: Delete completed activities every 60s instead of immediately
2. **Remove heartbeats**: For activities <5 minutes, trust workers or use shorter timeouts
3. **Soft delete**: Mark as deleted, cleanup in background

**Expected Impact**: 2-5% improvement

## Recommendations

### Tier 1: Critical Fixes (Required for MVP)

1. **Fix Parallel Workflow Failures** ⏱️ 2-4 hours
   - Add debug logging to orchestrator scheduling
   - Verify all parallel activities scheduled simultaneously
   - Test with increased logging

2. **Reduce Event Polling Backoff** ⏱️ 15 minutes
   ```rust
   AdaptiveBackoff::new(
       Duration::from_millis(1),    // Min: 1ms (was 10ms)
       Duration::from_millis(100),  // Max: 100ms (was 5000ms)
       1.2,                         // Slower growth
   )
   ```

3. **Increase Worker Polling Frequency** ⏱️ 15 minutes
   - Reduce worker backoff from 10ms-5s to 1ms-500ms
   - Or implement long-polling on worker poll endpoint

### Tier 2: Performance Optimizations

4. **Batch Activity Deletion** ⏱️ 30 minutes
   - Delete completed activities every 60s in background task
   - Saves 1-6% of execution time

5. **Remove Heartbeat Updates** ⏱️ 20 minutes
   - For activities <5 minutes, skip heartbeat updates
   - Saves 1-3% of execution time

6. **Add Event Cleanup** ⏱️ 1 hour
   - Archive or delete events >24 hours old
   - Keeps event table small for faster queries

### Tier 3: Architectural Improvements

7. **Implement Event Batching** ⏱️ 2-3 hours
   - Read up to 100 events per poll (currently batched already)
   - Process multiple events in single orchestrator cycle

8. **Add Connection Pool Monitoring** ⏱️ 1 hour
   - Log connection pool stats (active, idle, waiting)
   - Detect pool exhaustion

9. **Long-Polling for Workers** ⏱️ 3-4 hours
   - Workers wait up to 30s for activity instead of immediate 404
   - Reduces HTTP request overhead
