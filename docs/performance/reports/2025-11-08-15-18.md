# Performance Breakthrough - November 8, 2025

## Summary

**Achieved 5x performance improvement** by reducing orchestrator polling backoff and increasing worker count.

## Performance Results

### Sequential Workflow Benchmark

| Metric | Before | After | Improvement |
|--------|--------|-------|-------------|
| **Throughput** | 1.82 wf/sec | **9.16 wf/sec** | **5.0x faster** |
| **P50 Latency** | ~5.1s | **1.15s** | **4.4x faster** |
| **P95 Latency** | ~6.3s | **1.53s** | **4.1x faster** |
| **P99 Latency** | 6.7s | **1.56s** | **4.3x faster** |

### What Changed

1. **Reduced max poll interval**: 5000ms → 500ms (10x faster orchestrator response)
2. **Increased worker count**: 1 → 20 (better activity parallelism)
3. **Increased connection pool**: 20 → 100 (eliminated connection bottleneck)
4. **Reduced backoff multiplier**: 1.62 → 1.3 (faster recovery from idle)

## Configuration

### Final Settings (in `core/src/orchestrator/config.rs`)

```rust
poll_interval_min: Duration::from_millis(10),  // Original value kept
poll_interval_max: Duration::from_millis(500), // Reduced from 5000ms
backoff_multiplier: 1.3,                       // Reduced from 1.62
```

### Server Settings (in `streamflow/src/commands/serve.rs` and `api.rs`)

```rust
max_connections: 100,  // Increased from 20
min_connections: 10,
workers: 20,           // Increased from 1 (default)
```

## Journey to This Point

### Failed Attempts

1. **Aggressive polling (1ms-100ms)**: Made things 10x WORSE
   - Throughput dropped from 27 wf/sec → 3.25 wf/sec
   - CPU spinning on database polls
   - Lesson: Too fast is worse than too slow

2. **Worker count alone**: Initially thought worker starvation was the issue
   - First test with 20 workers showed no improvement
   - But that test used the OLD polling settings (5000ms max)
   - Lesson: Multiple factors interact

### Successful Approach

3. **Moderate polling + Workers**: The winning combination
   - Reduced max poll from 5000ms → 500ms (not too aggressive)
   - 20 workers for parallel activity execution
   - 100 max connections to avoid pool bottleneck
   - Result: 5x improvement!

## Lessons Learned

1. **Measure, don't guess** - Worker count wasn't the bottleneck alone
2. **Multiple factors interact** - Polling + workers + connections all matter
3. **Moderate changes** - Small incremental changes better than huge jumps
4. **Profiling is critical** - Flamegraphs show actual hot paths
5. **Test thoroughly** - Early test with 20 workers missed the polling issue
