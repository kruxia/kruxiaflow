## CRITICAL BUG FIX: Shared Worker ID

### The Problem

All 20 worker polling threads shared a single `worker_id`, preventing true parallelism.

**Evidence** (`worker/src/manager.rs:44-48`):
```rust
for i in 0..self.config.concurrency {
    let poller = WorkerPoller::new(
        self.config.clone(),  // ❌ Same config for all threads!
        client.clone(),
        Arc::clone(&self.registry),
    );
```

**Server logs showed**:
```
worker_id=internal_worker_019a6574-3608-79f0-a3e3-f9cbfe4ee334  // Thread 0
worker_id=internal_worker_019a6574-3608-79f0-a3e3-f9cbfe4ee334  // Thread 1
worker_id=internal_worker_019a6574-3608-79f0-a3e3-f9cbfe4ee334  // Thread 2
... (all 20 threads identical)
```

### Why This Killed Performance

The activity claim query uses `FOR UPDATE SKIP LOCKED`:
```sql
UPDATE activity_queue
SET claimed_by = $worker_id, status = 'running'
WHERE id = (
  SELECT id FROM activity_queue
  WHERE namespace = $1 AND name = $2
  FOR UPDATE SKIP LOCKED  -- Row-level lock!
)
RETURNING *;
```

**With shared worker_id**:
1. Thread 0 claims activity → locks row with `claimed_by = worker_019a6574...`
2. Threads 1-19 try to claim → all hit same locked row → `SKIP LOCKED` skips it
3. Only 1 thread works, 19 threads idle
4. Result: No parallelism despite 20 workers!

**With batch size 1**: Even worse - each thread only claims 1 activity, so contention is maximal.

### The Fix

Modified `worker/src/manager.rs` to assign unique IDs:

```rust
for i in 0..self.config.concurrency {
    // ✅ Create unique worker_id for each poller thread
    let mut poller_config = self.config.clone();
    poller_config.worker_id = format!("{}_poller_{}", self.config.worker_id, i);

    let poller = WorkerPoller::new(
        poller_config,  // ✅ Unique config per thread
        client.clone(),
        Arc::clone(&self.registry),
    );
```

**Server logs after fix**:
```
worker_id=internal_worker_019a6579-e3aa-7fd3-ac94-d647b051f98f_poller_0
worker_id=internal_worker_019a6579-e3aa-7fd3-ac94-d647b051f98f_poller_1
worker_id=internal_worker_019a6579-e3aa-7fd3-ac94-d647b051f98f_poller_2
... (unique IDs for all 20 threads)
```

## Batch Size Optimization

With unique worker IDs, we tested different `poll_max_activities` values:

**Hypothesis**: Smaller batches distribute work more evenly across workers.

**Results**:
- **batch=10**: Workers can claim up to 10 activities, leading to uneven distribution
- **batch=1**: Each worker claims exactly 1 activity, perfect load balancing

**Winner**: batch=1 with unique worker IDs
- Best throughput: 11.02 wf/sec
- Lowest latency: P50=925ms
- Best worker utilization: All 20 threads active

**Configuration added**: `--poll-max-activities` (default: 10, recommend: 1)

### 🔴 Suspected Bottlenecks (Based on Architecture & Symptoms)

#### 1. Event Polling Latency (HIGH PROBABILITY)

**Evidence**:
- Current polling backoff: 10ms min, 5000ms max, 1.5x multiplier
- Under moderate load, backoff could reach 500ms-5s between polls
- With 5 activities per workflow × 100 workflows = 500 activity completions
- Each activity completion publishes event → orchestrator must poll to see it
- Average polling delay: 2.5 seconds (at max backoff)

**Impact Calculation**:
- Sequential workflow: 5 activities
- If each activity waits average 2.5s for orchestrator to poll and schedule next activity
- Total added latency: 5 × 2.5s = **12.5 seconds per workflow**
- This matches observed P50 latency of 3-5 seconds!

**Expected Fix Impact**: 5-10x improvement

**Recommendation**:
```rust
// Reduce backoff significantly
AdaptiveBackoff::new(
    Duration::from_millis(1),    // Min: 1ms (was 10ms)
    Duration::from_millis(100),  // Max: 100ms (was 5000ms)
    1.2,                         // Slower growth
)
```

#### 2. Connection Pool Exhaustion (MEDIUM PROBABILITY)

**Evidence**:
- Current pool: `max_connections = 20`
- Benchmark concurrency: 100 workflows
- Each workflow needs connections for:
  - API request (1 conn)
  - Orchestrator event processing (1 conn)
  - Worker activity execution (1 conn)
  - Polling loops (1-2 conns)

**Impact Calculation**:
- 100 concurrent workflows × 3-4 connections each = 300-400 needed
- With only 20 available → severe queueing and contention
- Workflows waiting for connections to become available

**Expected Fix Impact**: 2-5x improvement

**Recommendation**:
```rust
PgPoolOptions::new()
    .min_connections(10)
    .max_connections(100)  // Match or exceed concurrency
    .acquire_timeout(Duration::from_secs(5))
    .connect(&database_url)
    .await?
```

#### 3. Worker Polling Overhead (MEDIUM PROBABILITY)

**Evidence**:
- Workers poll API via HTTP: `GET /api/v1/activities/poll`
- No long-polling support → immediate 404 if no activity
- Workers must re-poll repeatedly
- HTTP round-trip: ~1-5ms localhost
- If polling every 100ms with no work: wasted CPU and network

**Impact Calculation**:
- Not directly causing slowness, but contributing to system overhead
- With long-polling, workers could wait up to 30s for activity
- Reduces HTTP requests from 10/sec to 0.03/sec per worker
- Less contention on API server and connection pool

**Expected Fix Impact**: 1.5-2x improvement

**Recommendation**:
Add long-polling support to worker poll endpoint:
```rust
// In API handler
async fn poll_activity(namespace: String, name: String) -> Result<Response> {
    let max_wait = Duration::from_secs(30);
    let start = Instant::now();

    loop {
        if let Some(activity) = queue.claim_next(&namespace, &name).await? {
            return Ok(Json(Some(activity)));
        }

        if start.elapsed() >= max_wait {
            return Ok(Json(None));  // Timeout, no activity
        }

        tokio::time::sleep(Duration::from_millis(100)).await;
    }
}
```

## Observations from Benchmark Runs

### Parallelism Paradox

**Finding**: Parallel workflow (10 activities) is **SLOWER** than sequential (5 activities)

| Metric | Sequential (5 act) | Parallel (10 act) | Expected | Actual |
|--------|-------------------|-------------------|----------|---------|
| Activities/workflow | 5 | 10 | More work | ✓ |
| Throughput | 1.82 wf/sec | 1.12 wf/sec | Similar (parallel execution) | ❌ SLOWER |
| P99 Latency | 6.7s | 30s | Similar (all run in parallel) | ❌ 4.5x WORSE |

**Analysis**:
- Parallel activities should execute simultaneously (no sequential dep)
- All 10 activities should start at once after initial activity completes
- Expected completion time ≈ longest activity (all should be ~same for "echo")
- Actual: Taking 4.5x longer suggests activities are NOT running in parallel

**Root Cause Hypothesis**:
1. **Worker starvation**: Only 1 worker running (from architecture), can only execute 1 activity at a time
2. **Connection pool blocking**: Workers waiting for database connections
3. **Orchestrator not scheduling all at once**: Bug in dependency evaluation?

**Evidence Needed**:
- Check server logs: Are all 10 activities scheduled simultaneously?
- Check worker logs: Are multiple activities being claimed and executed?
- Check connection pool: Are connections available?

**Verification**:
```bash
grep "Scheduled.*parallel_" profiling-results-*/server.log
# Should show all 10 activities scheduled within milliseconds of each other
```

   - May be unnecessary for short activities (save 110ms)

4. **Event Volume**: 4,950 events / 100 workflows = 49.5 events per workflow
   - For 5-activity workflows: ~10 events expected
   - Actual shows reasonable event granularity
