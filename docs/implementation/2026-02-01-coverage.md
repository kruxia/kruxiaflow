### Testing Strategy for CLI Commands

CLI commands are challenging to unit test because they orchestrate I/O (database connections, server startup, HTTP calls). The approach:

1. **Extract testable logic** ŌĆö factor pure validation, configuration assembly, and output formatting into separate functions that can be tested in isolation
2. **Test validation paths** ŌĆö parameter validation, environment variable parsing, configuration merging
3. **Test output formatting** ŌĆö status display, health check formatting, error messages
4. **Mock-based tests** ŌĆö where feasible, test command logic against mocked services

## Implementation Priorities

### Ordering Rationale

1. **Phase 0** first ŌĆö immediate coverage boost with no code changes
2. **Phase 1** next ŌĆö orchestrator is the most critical component
3. **Phase 2** follows ŌĆö worker activities are the second most critical
4. **Phase 3-4** can proceed in parallel ŌĆö API and CLI are independent
5. **Phase 5-6** are cleanup passes

### Testing Approach

- **Unit tests** for pure logic (validation, formatting, computation)
- **Mock-based tests** for service interactions (use `mockall` or trait objects)
- **Integration tests** for database-dependent code (use local Docker PostgreSQL)
- **No external service dependencies** (per project guidelines)
- Tests must not depend on external services but only on internal services running in the local Docker cluster

### Test Naming Convention

```
test_<component>_<scenario>_<expected_outcome>
```

Examples:
- `test_orchestrator_cancels_running_workflow`
- `test_llm_activity_maps_rate_limit_to_transient_error`
- `test_signal_handler_rejects_unauthenticated_request`
