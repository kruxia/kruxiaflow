# StreamFlow v0.2 Implementation Notes

**Version**: 0.2 (Revised)  
**Date**: October 25, 2025  
**Status**: Planning  
**Target Release**: Q1 2026

---

## Executive Summary

StreamFlow v0.1 revealed two critical issues that must be addressed for market success:

1. **Performance Problem**: Event streaming with parallel execution provided no ordering guarantees, resulting in worse performance than Temporal (the opposite of our goal)
2. **Developer Experience Problem**: Direct state machine definitions are not developer-friendly compared to market alternatives (Airflow DAGs, Temporal workflows)

**StreamFlow v0.2 Core Changes**:

- **Event-Driven Orchestration**: Activities scheduled only when workflow state indicates readiness
- **YAML DSL**: Declarative workflow definitions covering 70-80% of use cases
- **Programmatic Definition**: Python/JavaScript/Go that **compiles to YAML** (15-25% of use cases)
- **PostgreSQL Optimization**: Target >1,000 workflows/sec (10x competitor baseline)
- **Preserved Differentiators**: Single binary, AI-native features, cost intelligence, edge deployment

**Key Architectural Insight**: Python and JavaScript are **workflow definition languages** that compile to YAML at deployment time, not runtime decision engines. This eliminates external worker complexity and latency entirely.

## 1. Core Architecture

### 1.1 Execution Model: Event-Driven Dynamic Scheduling

**Key Principle**: Activities are only scheduled when the orchestrator determines they are ready based on current workflow state.

```
┌────────────────────────────────────────────────────┐
│ Orchestrator (Event-Driven)                       │
│                                                    │
│ 1. Subscribe to workflow state events             │
│ 2. On workflow created/activity completed:        │
│    ├── Load workflow definition                    │
│    ├── Evaluate current state                       │
│    ├── Determine which activities are NOW ready    │
│    └── Schedule ready activities to queue           │
│ 3. Publish updated workflow state event           │
└────────────────────────────────────────────────────┘
                         │
                         ↓
┌────────────────────────────────────────────────────┐
│ Activity Queue (PostgreSQL)                        │
│                                                    │
│ - Simple FIFO queue per activity type             │
│ - Activities added only when ready to execute     │
│ - No complex partitioning needed                  │
│ - Idempotent scheduling (prevent duplicates)      │
└────────────────────────────────────────────────────┘
                         │
                         ↓
┌────────────────────────────────────────────────────┐
│ Activity Workers                                   │
│                                                    │
│ 1. Poll activity queue by type                    │
│ 2. Execute activity logic                         │
│ 3. Publish result event                           │
│ 4. → Triggers orchestrator re-evaluation          │
└────────────────────────────────────────────────────┘
```

### 1.2 Multi-Syntax Workflow Definition

**Core Innovation**: All workflow syntaxes compile to the same internal DAG representation.

```
┌──────────────────────────────────────────────────┐
│ Workflow Definition (Multiple Syntaxes)          │
│                                                  │
│  YAML DSL        Python Builder    JavaScript    │
│     │                 │                  │       │
│     │                 │                  │       │
│     └─────────────────┴──────────────────┘       │
│                       │                          │
│                       ↓                          │
│              Internal DAG Representation         │
│              (activities + edges + conditions)   │
│                       │                          │
│                       ↓                          │
│         Orchestrator (Event-Driven Execution)    │
│          - Same <1ms evaluation performance      │
│          - No external decision workers          │
│          - No RPC latency                        │
└──────────────────────────────────────────────────┘
```

**Key Point**: Python/JavaScript code runs at **deployment time** (compilation), not at **runtime** (execution).

### 1.3 How Sequential Execution Works

**Example Workflow**:
```yaml
workflow: payment_processing
activities:
  - key: validate_payment
    namespace: payments
    following:
      - activity_key: authorize_card

  - key: authorize_card
    namespace: payments
    parameters:
      card_token: "{{validate_payment.card_token}}"
    following:
      - activity_key: capture_payment

  - key: capture_payment
    namespace: payments
    parameters:
      authorization_id: "{{authorize_card.authorization_id}}"
```

**Execution Timeline**:

```
t=0: Workflow starts
     Orchestrator evaluates:
       - validate_payment has no preceding activities → READY
       - authorize_card needs validate_payment → NOT READY
       - capture_payment needs authorize_card → NOT READY
     Action: Schedule validate_payment to queue

t=1: validate_payment completes
     Orchestrator re-evaluates:
       - validate_payment: done ✓
       - authorize_card needs validate_payment (done) → NOW READY
       - capture_payment needs authorize_card (not done) → NOT READY
     Action: Schedule authorize_card to queue

t=2: authorize_card completes
     Orchestrator re-evaluates:
       - authorize_card: done ✓
       - capture_payment needs authorize_card (done) → NOW READY
     Action: Schedule capture_payment to queue

t=3: capture_payment completes
     Orchestrator evaluates:
       - All activities complete → Workflow complete
     Action: Publish workflow completed event
```

**Key Insight**: Sequential execution is guaranteed because activities only enter the queue when their dependencies are satisfied. Workers cannot execute activities that aren't in the queue yet.

### 1.4 How Parallel Execution Works

**Example Workflow**:
```yaml
activities:
  - key: fetch_data
    following:
      - activity_key: analyze_security
      - activity_key: analyze_performance
      - activity_key: analyze_quality

  - key: analyze_security
    namespace: analysis
  
  - key: analyze_performance
    namespace: analysis
  
  - key: analyze_quality
    namespace: analysis

  - key: aggregate_results
    namespace: analysis
    parameters:
      security: "{{analyze_security.results}}"
      performance: "{{analyze_performance.results}}"
      quality: "{{analyze_quality.results}}"
    following:
      - activity_key: analyze_security
      - activity_key: analyze_performance
      - activity_key: analyze_quality
```

**Execution Timeline**:

```
t=0: Workflow starts
     Orchestrator evaluates:
       - fetch_data has no dependencies → READY
     Action: Schedule fetch_data

t=1: fetch_data completes
     Orchestrator re-evaluates:
       - All three analyze_* activities now have fetch_data complete → ALL READY
     Action: Schedule analyze_security, analyze_performance, analyze_quality
            (all three added to queue simultaneously)

t=2-4: Workers execute analyze_* activities in parallel (any order)

t=5: analyze_security completes
     Orchestrator re-evaluates:
       - aggregate_results needs all three analyses
       - analyze_security: done ✓
       - analyze_performance: not done yet
       - analyze_quality: not done yet
     Action: Nothing to schedule (waiting for others)

t=6: analyze_performance completes
     Orchestrator re-evaluates:
       - aggregate_results needs all three analyses
       - analyze_security: done ✓
       - analyze_performance: done ✓
       - analyze_quality: not done yet
     Action: Nothing to schedule (still waiting)

t=7: analyze_quality completes
     Orchestrator re-evaluates:
       - aggregate_results needs all three analyses
       - analyze_security: done ✓
       - analyze_performance: done ✓
       - analyze_quality: done ✓
     Action: Schedule aggregate_results
```

**Key Insight**: Parallel execution happens naturally when multiple activities become ready simultaneously. The join point (aggregate_results) only becomes ready when ALL dependencies are satisfied.

### 1.5 Activity Queue Schema

```sql
CREATE TABLE activity_queue (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    workflow_id UUID NOT NULL,
    activity_key TEXT NOT NULL,
    namespace TEXT NOT NULL,
    name TEXT NOT NULL,
    parameters JSONB NOT NULL,
    settings JSONB,
    status TEXT NOT NULL DEFAULT 'pending',
    scheduled_for TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    claimed_by UUID,
    claimed_at TIMESTAMPTZ,
    completed_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    -- Prevent duplicate scheduling
    UNIQUE(workflow_id, activity_key)
);

-- Index for worker polling by activity type
CREATE INDEX idx_queue_pending 
ON activity_queue(namespace, name, scheduled_for)
WHERE status = 'pending' 
  AND scheduled_for <= NOW();

-- Index for workflow status queries
CREATE INDEX idx_queue_workflow 
ON activity_queue(workflow_id, status);
```

**Worker Polling Query**:
```sql
-- Workers poll by activity type (namespace.name)
-- Uses FOR UPDATE SKIP LOCKED for concurrent worker safety
SELECT id, workflow_id, parameters, settings
FROM activity_queue
WHERE status = 'pending'
  AND scheduled_for <= NOW()
  AND namespace = $1
  AND name = $2
ORDER BY scheduled_for ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

### 1.6 Orchestrator Evaluation Logic

**Readiness Evaluation**:

```rust
fn evaluate_workflow_readiness(
    workflow_def: &WorkflowDefinition,
    workflow_state: &WorkflowState,
) -> Vec<ActivityToSchedule> {
    let mut ready_activities = Vec::new();
    
    for activity in &workflow_def.activities {
        // Skip if already completed or currently in queue
        if workflow_state.is_activity_done(&activity.key) 
           || workflow_state.is_activity_queued(&activity.key) {
            continue;
        }
        
        // Check all preceding edges (fan-in dependencies)
        let mut all_preceding_ready = true;
        for edge in activity.edges.iter().filter(|e| e.preceding_key.is_some()) {
            let preceding = edge.preceding_key.as_ref().unwrap();
            
            // Must be completed
            if !workflow_state.is_activity_done(preceding) {
                all_preceding_ready = false;
                break;
            }
            
            // Check edge conditions if specified
            if let Some(conditions) = &edge.conditions {
                if !conditions.evaluate(workflow_state) {
                    all_preceding_ready = false;
                    break;
                }
            }
        }
        
        if all_preceding_ready {
            ready_activities.push(activity.clone());
        }
    }
    
    ready_activities
}
```

**Event-Driven Orchestration Loop**:

```rust
async fn handle_workflow_event(event: WorkflowEvent) {
    match event.event_type {
        WorkflowEventType::Created => {
            // Load workflow definition
            let workflow_def = load_definition(&event.workflow_type).await?;
            
            // Initial evaluation - find activities with no dependencies
            let ready = evaluate_workflow_readiness(&workflow_def, &event.state);
            
            // Schedule to queue
            schedule_activities(event.workflow_id, ready).await?;
        }
        
        WorkflowEventType::ActivityCompleted { activity_key, result } => {
            // Update workflow state
            update_workflow_state(
                event.workflow_id,
                activity_key,
                ActivityStatus::Completed,
                result
            ).await?;
            
            // Re-evaluate what's now ready
            let workflow_state = get_workflow_state(event.workflow_id).await?;
            let workflow_def = load_definition(&workflow_state.workflow_type).await?;
            let ready = evaluate_workflow_readiness(&workflow_def, &workflow_state);
            
            // Schedule newly ready activities
            schedule_activities(event.workflow_id, ready).await?;
            
            // Check if workflow complete
            if workflow_state.is_complete(&workflow_def) {
                publish_event(WorkflowEvent {
                    workflow_id: event.workflow_id,
                    event_type: WorkflowEventType::Completed {
                        output: workflow_state.output,
                    },
                }).await?;
            }
        }
        
        WorkflowEventType::ActivityFailed { activity_key, error } => {
            // Handle retry logic or workflow failure
            handle_activity_failure(event.workflow_id, activity_key, error).await?;
        }
    }
}
```

**Idempotency Guarantee**:

```rust
async fn schedule_activities(
    workflow_id: Uuid,
    activities: Vec<Activity>,
) -> Result<()> {
    for activity in activities {
        // ON CONFLICT DO NOTHING prevents duplicate scheduling
        sqlx::query!(
            r#"
            INSERT INTO activity_queue 
                (workflow_id, activity_key, namespace, name, parameters, settings)
            VALUES ($1, $2, $3, $4, $5, $6)
            ON CONFLICT (workflow_id, activity_key) DO NOTHING
            "#,
            workflow_id,
            activity.key,
            activity.namespace,
            activity.name,
            activity.parameters,
            activity.settings,
        )
        .execute(&pool)
        .await?;
    }
    Ok(())
}
```

## 2. Workflow Definition: YAML DSL

### 2.1 Design Philosophy

**Goals**:
- 70-80% of workflows expressible in declarative YAML
- Familiar syntax (similar to Airflow DAGs, GitHub Actions)
- Type-safe (validated at load time)
- Easy to visualize and understand

### 2.2 Core YAML Syntax

**Minimal Example**:

**Sequential Workflow**:

### 2.3 Edge Definition: following_key (Primary)

**Recommendation**: Use `following_key` as the primary approach, with `preceding_key` available for multi-input joins.

**following_key (Fan-Out)**:
```yaml
# One activity leads to multiple next activities
- key: fetch_data
  namespace: data
  following:
    - activity_key: analyze_security
    - activity_key: analyze_performance
    - activity_key: analyze_quality
```

**preceding_key (Fan-In/Join)**:
```yaml
# Multiple activities converge to one next activity
- key: aggregate_results
  namespace: analysis
  following:
    - activity_key: analyze_security
    - activity_key: analyze_performance
    - activity_key: analyze_quality
```

**Why following_key is Primary**:
1. ✅ Natural sequential thinking: "after this, do that"
2. ✅ Matches Airflow's `>>` operator (market leader)
3. ✅ Easier to read top-to-bottom
4. ✅ Better for visualization (arrows point forward)
5. ✅ Most workflows are sequential or fan-out

**When to use preceding_key**:
- Multi-input joins (parallel activities converging)
- When dependency relationships are clearer from the dependent side

## 3. Programmatic Workflow Definition (Python/JavaScript)

### 3.1 Design Philosophy

**Key Principle**: Python, JavaScript, and Go are **workflow definition languages** that compile to YAML/DAG at deployment time, not runtime decision engines.

```
Deployment Time:                Runtime:
────────────────                ────────
Python code executes    →       Compiled YAML loaded
  ↓                            ↓
Builder API constructs DAG     Orchestrator executes
  ↓                            ↓
Validates & serializes         <1ms decision latency
  ↓                            (same as YAML)
Saves as YAML
```

**Benefits**:
- ✅ More convenient and less verbose workflow definition in familiar programming language
- ✅ Co-location with custom activity implementations
- ✅ Type checking and IDE support
- ✅ Use language features (loops, conditionals) at **definition time** for convenience
- ✅ Same <1ms performance as YAML workflows (compiles to same YAML format)
- ✅ No RPC latency
- ✅ No external decision workers
- ✅ Simpler architecture

### 3.2 When to Use Each Syntax

**YAML DSL** (70-80% of workflows):
- Simple sequential/parallel patterns
- Workflows defined by non-developers
- Static workflow structure
- Fast prototyping

**Python/JavaScript Builder** (15-25% of workflows):
- More convenient and less verbose workflow definition in familiar programming language
- Co-location with custom activity implementations
- Type checking and IDE autocomplete
- Reusable workflow components and patterns
- Build-time generation of workflow structures (e.g., create N parallel activities based on configuration)
- Note: Provides same runtime capabilities as YAML, compiles to YAML before execution

**Rust (internal)** (0-5% of workflows):
- Performance-critical workflows
- Core platform workflows

### 3.3 Python Builder API

**Simple Sequential Workflow**:

```python
from streamflow import Workflow, Activity

# Create workflow
workflow = Workflow("payment_processing", version="1.0")

# Define activities with fluent API
validate = Activity("validate_payment", namespace="payments") \
    .with_parameters(
        card_token=workflow.input.card_token,
        amount=workflow.input.amount
    ) \
    .with_outputs("valid", "card_last4")

authorize = Activity("authorize_card", namespace="payments") \
    .with_parameters(
        card_token=workflow.input.card_token,
        amount=workflow.input.amount
    ) \
    .with_outputs("authorization_id", "authorized") \
    .when(validate.outputs.valid == True) \
    .after(validate)

capture = Activity("capture_payment", namespace="payments") \
    .with_parameters(
        authorization_id=authorize.outputs.authorization_id
    ) \
    .with_outputs("transaction_id") \
    .when(authorize.outputs.authorized == True) \
    .after(authorize)

# Add activities to workflow
workflow.add_activities(validate, authorize, capture)

# Deploy (compiles to YAML at deployment time)
workflow.deploy(environment="production")
```

**What Happens at Deployment**:
1. Python code executes locally or in CI/CD
2. Builder API constructs internal DAG representation
3. DAG validates (edges valid, no cycles, etc.)
4. DAG serializes to YAML format
5. YAML uploaded to workflow registry
6. Done - Python doesn't run again

**Generated YAML** (what orchestrator sees):
```yaml
workflow: payment_processing
version: "1.0"

activities:
  - key: validate_payment
    namespace: payments
    parameters:
```

```rust
// Batch activity scheduling
async fn schedule_activities_batch(
    pool: &PgPool,
    workflow_id: Uuid,
    activities: Vec<Activity>,
) -> Result<()> {
    let mut tx = pool.begin().await?;
    
    for chunk in activities.chunks(100) {
        let mut query_builder = QueryBuilder::new(
            "INSERT INTO activity_queue 
             (workflow_id, activity_key, namespace, name, parameters) "
        );
        
        query_builder.push_values(chunk, |mut b, activity| {
            b.push_bind(workflow_id)
             .push_bind(&activity.key)
             .push_bind(&activity.namespace)
             .push_bind(&activity.name)
             .push_bind(&activity.parameters);
        });
        
        query_builder.push(" ON CONFLICT (workflow_id, activity_key) DO NOTHING");
        
        query_builder.build().execute(&mut *tx).await?;
    }
    
    tx.commit().await?;
    Ok(())
}
```

```rust
async fn check_budget_before_schedule(
    workflow_id: Uuid,
    activity_budget: Option<f64>,
) -> Result<()> {
    let workflow_spent = get_workflow_cost(workflow_id).await?;
    let workflow_budget = get_workflow_budget(workflow_id).await?;
    
    if workflow_spent >= workflow_budget {
        return Err(Error::BudgetExceeded {
            workflow_id,
            budget: workflow_budget,
            spent: workflow_spent,
        });

        let remaining = workflow_budget - workflow_spent;
        if activity_limit > remaining {
            return Err(Error::ActivityBudgetExceedsRemaining {
                activity_limit,
                remaining,
            });
        }
    }
    
    Ok(())
}
```

```rust
// On workflow replay after crash
if activity.settings.deterministic == false {
    // Check if this activity already executed before crash
    if let Some(previous_result) = get_previous_result(activity_id).await? {
        // Use previous result instead of re-executing
        return Ok(previous_result);
```

## 10. Open Questions & Design Decisions

### 10.1 Edge Definition: Following vs Preceding (RESOLVED)

**Decision**: Use `following_key` as primary with `preceding_key` available for joins.

**Rationale**:
- Natural sequential thinking ("after this, do that")
- Matches Airflow's dominant >> operator pattern
- Easier to read top-to-bottom in YAML
- `preceding_key` available for multi-input fan-in patterns

### 10.2 State Management

**Question**: How should workflow state be modified?

**Options**:
A. Only via activities (pure functional workflows)
B. YAML state updates (limited, declarative) ← **RECOMMENDED**
C. Full mutable state (like Temporal)

**Recommendation**: Option B.

```yaml
- key: update_iteration
  namespace: streamflow_core
  name: state_update
  parameters:
    updates:
      - increment: iteration
      - set:
          last_search_time: "{{NOW}}"
      - append_to: search_history
        value: "{{current_search.results}}"
```

**Rationale**: Declarative state updates keep workflows pure while allowing common patterns like iteration counts and context accumulation.

### 10.3 Workflow Versioning

**Question**: How to handle workflow definition changes for running workflows?

**Options**:
A. Immutable workflows (version in name) ← **RECOMMENDED for MVP**
B. Temporal-style determinism checks
C. Automatic migrations

**Recommendation**: Option A for V2, evaluate B for V3.

**Implementation**:
```yaml
workflow: payment_processing_v1
version: "1.0"
```

**Behavior**:
- Running workflows continue with version they started with
- New workflow starts use latest version
- Version history maintained in database

**Future Enhancement**: Add determinism validation (workflow definition changes that don't affect running instances).

### 10.4 Expression Language Scope

**Question**: How much logic should YAML expressions support?

**Options**:
A. Minimal (just field access and comparison)
B. Moderate (conditions, simple arithmetic) ← **RECOMMENDED**
C. Maximal (loops, functions, complex logic)

**Recommendation**: Option B.

**Supported**:
- Field access: `{{activity.field}}`, `{{ARG.nested.field}}`
- Comparison: `==`, `!=`, `<`, `>`, `<=`, `>=`
- Logical: `AND`, `OR`, `NOT`
- Simple arithmetic: `+`, `-`, `*`, `/`
- String interpolation: `"Amount: ${{amount}}"`

**Not Supported** (use Python/JavaScript workflows):
- Custom functions
- Complex algorithms
- Iteration/loops in expressions
- Regular expressions

**Rationale**: Keep YAML readable. Push complex logic to Python/JavaScript where proper programming languages excel.

### 11.1 Performance Risk

**Risk**: May not achieve 1,000 workflows/sec target on PostgreSQL.

**Mitigation**:
- Early performance benchmarking (Phase 1)
- Incremental optimization based on profiling
- Fallback: Lower target to 500 workflows/sec (still 5x better than competitors)
- Alternative: Add optional Redis layer for hot paths

### 11.2 Adoption Risk: YAML vs Code

**Risk**: Developers may prefer code-based workflow definitions over YAML.

**Mitigation**:
- Provide Python/JavaScript from day one
- Show clear migration path from Temporal/Airflow
- Emphasize YAML is optional (code works too)
- Publish comparison showing YAML is 86% more concise
- Python/JavaScript compiles to same performance as YAML

### 11.3 Compilation Complexity Risk

**Risk**: Python/JavaScript → YAML compilation may be complex or buggy.

**Mitigation**:
- Start with simple Builder API (fluent interface)
- Comprehensive test suite for compilation
- Clear error messages with source locations
- Validate generated YAML before deployment
- Document escape hatches for edge cases

### 11.4 Market Timing Risk

**Risk**: Competitors may add similar features before launch.

**Mitigation**:
- Fast MVP (16 weeks to functional v0.2)
- Focus on unique combinations (PostgreSQL + single binary + AI features)
- Build community early (open source from day one)
- Emphasize operational simplicity, not just features
